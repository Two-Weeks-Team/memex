01Real TurboQuant benchmark numbers in docs/benchmarks.md

issue #13

Current state

docs/benchmarks.md ships a table of recall@10 / p95 latency / disk size with the v3 production setting at 0.98–0.99 recall, ~5 ms p95, 12 % size. The doc itself flags these as illustrative reference values taken from Qdrant's own TurboQuant blog post on a comparable BGE-small fixture. They have never been measured on the memex corpus.

The current numbers below are placeholder reference values taken from the Qdrant team's own TurboQuant blog post on a comparable fixture (BGE-small, 1k-10k corpus). They are labeled illustrative in landing copy and should not be quoted as production measurements until the recipe in §3 has been executed against the actual Memex fixture and the table updated… — docs/benchmarks.md (committed in PR #12, cf412)

Why it wasn't done in PR #12

PR #12 REV-13 (CodeRabbit #6) discovered the original recipe was broken (cargo run -- eval ndcg was a non-existent binary; eval_ndcg is a library module exposing ndcg_at_10() + mean_ndcg_at_10()). The fix was an honest recipe rewrite, not real numbers — because:

  • Quant knobs live in src-tauri/src/schema.rs::quant_search(), hard-coded, no CLI flag.
  • A measurement requires source edits + cargo build + reindex (~10 min) per row.
  • No labeled-queries fixture exists in the repo; would have to be authored.

Doing all of that inside PR #12 would have ballooned scope by several days and is properly a separate feature PR.

Proposed solution

A new gated binary src-tauri/src/bin/bench.rs driven by:

memex bench quant-sweep \
  --corpus ~/.claude/projects/ \
  --queries fixtures/labeled-queries.jsonl \
  --configs f32,scalar,binary,tq-bits2,tq-bits1 \
  --output docs/benchmarks-measured.json

Each config is materialised by a new schema::quant_config_for(QuantMode) that returns the matching QuantizationConfig instead of the hardcoded one. The binary drops + recreates the collection per config, indexes, queries, captures p95 from tokio::time::Instant deltas. An xtask render-bench post-step renders into the existing markdown table format.

Acceptance criteria (excerpt)

  • cargo build --features bench produces target/release/bench
  • fixtures/labeled-queries.jsonl checked in (≥20 queries × relevance labels)
  • docs/benchmarks.md table shows 5 measured rows with reproducible recipe
  • Regression test in src-tauri/tests/bench_e2e.rs sanity-checks ndcg > 0.5
  • wired-but-dormant.md §C: remove "benchmarks pending"

Why issue, not blocker

PR #12's claims survive on the "illustrative" caveat — acceptable while the table is explicitly labeled illustrative. The work is mechanical and well-defined; no architecture decisions outstanding. Gating criterion is "the table shows measured numbers", not any external date.

02Custom tokenizer for ai_title (camelCase / snake_case)

issue #14

Current state

ai_title is indexed with a Text payload index using the default word tokenizer (declared in src-tauri/src/schema.rs). For natural-language titles like "Debugging the auth flow" this is correct. For identifier-rich titles like "getUserData refactor" or "fix_dns_resolver bug", it silently fails: getUserData remains one token, so a search for user does not match it.

Confirmed against Qdrant 1.18 official documentation

Per the Qdrant 1.18 Indexing → Tokenizers page (re-verified 2026-05-28), Qdrant ships exactly 4 built-in tokenizers:

TokenizerBehaviorSplits getUserData?
word (default)space + punctuation + special charsNo (1 token)
whitespacespace onlyNo (1 token)
prefixword + prefix index of each tokenNo (still 1 token + its prefixes)
multilingualcharabia + vaporetto for non-Latin scriptsNo (designed for natural language, not identifiers)

The Rust SDK 1.18's TextIndexParamsBuilder only accepts the TokenizerType enum (Word | Whitespace | Prefix | Multilingual). Custom tokenizer registration is not exposed in any public API — neither REST nor gRPC. Qdrant's public issue tracker as of today has no accepted proposal for a custom-tokenizer feature either.

Solution — client-side identifier expansion (single answer, not a workaround)

Solving this in Qdrant itself (upstream feature request or fork) was rejected: we control every surface that touches ai_title — the tokenizer layer being inside or outside Qdrant gives the same retrieval result, but client-side keeps us self-contained, ships today, and avoids a future re-migration when/if upstream catches up.

Client-side payload expansion

Expand identifiers at index time into a second indexed field ai_title_tokens (same word tokenizer applies — once split client-side, word is correct). Original ai_title preserved for display. 

// src-tauri/src/indexer.rs
fn expand_identifiers(s: &str) -> String {
    // "getUserData fix_dns_bug" →
    // "getUserData get User Data fix_dns_bug fix dns bug"
    ...
}

// at point build:
payload.insert("ai_title",        title.clone());
payload.insert("ai_title_tokens", expand_identifiers(&title));
  • ✓ ~50 LOC + tests · zero Qdrant change · zero external dependency
  • ✓ Industry precedent: Elasticsearch word_delimiter_graph filter, GitHub Code Search
  • ~ +30 % payload bytes on the title field only (negligible against content bulk)
  • ~ Search-time field selection heuristic in web.rs/mcp.rs: route identifier-shaped queries to ai_title_tokens

Why not fork Qdrant or wait for 1.19+: we'd inherit security-update responsibility (fork) or carry the client-side helper anyway during a 6–12-month wait (upstream), then face a second migration to drop it. Single migration beats two.

03Widen indexer::LensWeights to mirror lens::LensWeights

issue #15

The actual gap (verified by reading both source files)

lens::LensWeights (internal, 8 fields)

pub struct LensWeights {
    pub content:      f32,
    pub tool:         f32, // reused for tool_sparse
    pub path:         f32, // reused for path_sparse
    pub error:        f32,
    pub code:         f32,
    pub content_late: f32,
    pub diversity:    Option<f32>,
    pub fusion:       FusionMode,
}

indexer::LensWeights (public API, 6 fields)

pub struct LensWeights {
    pub content:      f32,
    pub tool:         f32,
    pub path:         f32,
    pub error:        f32,
    pub code:         f32,
    pub content_late: f32,
    // ⚠️ NO diversity
    // ⚠️ NO fusion
}

The From<indexer::LensWeights> for lens::LensWeights impl at src-tauri/src/lens.rs:869 hardcodes the two missing fields: 

impl From<crate::indexer::LensWeights> for LensWeights {
    fn from(v: crate::indexer::LensWeights) -> Self {
        Self {
            content:      v.content,
            tool:         v.tool,
            path:         v.path,
            error:        v.error,
            code:         v.code,
            content_late: v.content_late,
            diversity:    None,              // ← silently dropped
            fusion:       FusionMode::Formula, // ← silently fixed
        }
    }
}

Real-world impact

Callers reaching memex through any external surface — Tauri commands, the /api/lens_search JSON endpoint, MCP — go through indexer::LensWeights. Today they cannot request:

  • MMR diversification (diversity: Some(d)): the “diverse results” UX knob is internal-only.
  • RRF fusion (fusion: FusionMode::Rrf): the alternative to formula-based fusion is internal-only.

Both are explicitly documented in docs/wired-but-dormant.md as wired:off (§B.1 RRF, §B.2 MMR) — i.e. the code path exists and works, only the public knob is missing.

Why this wasn't fixed in PR #12

PR #12 REV-8 (Codex P2 #17, CRITICAL) caught a related issue: the content_late serde default in indexer::LensWeights::default() was still 0.0 while the internal struct had been bumped to 0.25 by T3.3. That 6-field overlap was fixed. REV-15 (self-review) then noticed the additional field-set gap but flagged it as out-of-scope, since widening the public API surface requires regenerating TypeScript SDK bindings, adding two frontend controls, and writing a one-time migration note for JSON callers.

Proposed solution

Mirror the 2 missing fields with serde defaults that preserve current behavior:

pub struct LensWeights {
    // ... existing 6 fields ...
    #[serde(default)]
    pub diversity: Option<f32>,
    #[serde(default)]
    pub fusion: FusionMode,
}

And make the From impl lossless (round-trip property test).

The follow-up activates only if MCP grows a write surface — which is a much bigger product decision than a metric wire-up:
