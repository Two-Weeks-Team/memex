01Issue #13 — Real benchmark numbers for docs/benchmarks.md

issue #13

Restated problem

docs/benchmarks.md ships an illustrative table (TurboQuant bits-2 vs f32 baseline) sourced from Qdrant's own blog. Real numbers on the memex corpus do not exist. The blocker is not the measurement itself but the fact that quant knobs are hardcoded in schema.rs::quant_search() — switching modes today requires source-edit + rebuild + full reindex, which is ~15 min/row × 5 rows = 75 min just for one sweep.

Alternatives matrix

#OptionVerdictRationale
ANew gated bench binary with custom quant-sweep subcommand considered Reinvents what cargo bench + Criterion already does (timing, statistical analysis, regression detection). Custom CLI = custom output format ≠ industry tooling integration.
BCriterion-based benchmark (cargo bench) with sub-benchmarks per config picked (half) Rust-native. Built-in statistical noise filtering. JSON / HTML output. Already used in similar Qdrant client benches. Lets us write #[criterion::bench] for each (quant_mode, corpus_size) cell.
CRuntime-configurable quant mode via env var / config file (no rebuild) picked (other half) Removes the “rebuild between rows” cost. QuantizationConfig is data, not behavior — selecting it at runtime is a 30-LOC change to schema::quant_search() reading from MEMEX_QUANT_MODE.
DCI matrix: nightly benchmark across 5 configs, retain timeseries future work Valuable but out-of-scope. Needs durable storage (Grafana / BigQuery) + a stable benchmark machine. Add after B+C ship and produce a baseline.
EReuse qdrant/vector-db-benchmark (Python + gRPC) rejected Excellent suite but designed for comparing vector DBs, not for comparing quant knobs of one DB. Our fixture is the user's session corpus, not a public ANN dataset. Adapting it to BGE-small + our schema + our labeled-queries would be more work than B+C.
FProperty-based / fuzzed benchmark (varying corpus sizes) rejected Wrong tool. Property-based testing finds bugs; benchmarks measure performance. Sweeping corpus sizes is interesting but should be a parameter of B, not its own framework.

Industry precedent — Best practice

  • Criterion + iai-callgrind: the de-facto Rust-ecosystem pattern. Criterion handles wallclock + statistical analysis; iai handles instruction-counted comparisons for catching regressions hidden by noise.
  • RocksDB db_bench: single binary, JSON output, runtime flags to switch comparisons. Decades-proven design — exactly what we need for "compare 5 quant configs on same fixture".
  • Qdrant vector-db-benchmark: validates that “separate process drives the DB through its actual network API” is the right boundary for vector-DB benchmarking, not in-process calls. We follow the same boundary.
  • BEIR (Benchmarking IR): for retrieval-quality metrics, nDCG@10 with a labeled-queries fixture is the standard — exactly what eval_ndcg::mean_ndcg_at_10 already implements. We extend the fixture, not the metric.

Refined recommendation — B + C combined

Two small, independent changes that compose:

  1. Runtime quant config (C): read MEMEX_QUANT_MODE = f32 | scalar | binary | tq-bits2 | tq-bits1 at collection-create time inside schema::quant_search(). ~30 LOC + 5 tests.
  2. Criterion benchmark (B): new file src-tauri/benches/quant_sweep.rs that, for each mode in the env-var matrix, drops + recreates the collection, indexes the labeled corpus, runs the labeled queries, reports nDCG + p95. Single command: cargo bench --bench quant_sweep.

Output flows into Criterion's standard target/criterion/report/index.html. A small xtask render-bench-table reads Criterion's JSON output and overwrites the markdown table in docs/benchmarks.md. The illustrative caveat goes away on first run.

Why this beats option A: zero custom binary, zero rebuild-per-row, integrates with cargo bench regression detection out of the box. The labeled-queries fixture authoring (~20 queries × 5 relevant session-ids) remains the bulk of the work — but that's data, not code.

02Issue #14 — Identifier-aware tokenizer for ai_title

issue #14

Restated problem

Qdrant 1.18 ships 4 builtin tokenizers (word / whitespace / prefix / multilingual) and no public API for registering a custom one. Identifier-shaped titles like "getUserData refactor" tokenize as a single token, breaking partial-word matches.

Alternatives matrix — all client-side (upstream & fork rejected up-front)

Upstream Qdrant request and fork were both rejected: we control every surface touching ai_title; the tokenizer layer inside vs. outside Qdrant produces identical retrieval results; client-side avoids upstream timeline risk and double-migration cost.

#OptionVerdictRationale
AString concat: ai_title_tokens = "getUserData get User Data" stored as single string considered Original v1 recommendation. Works, but the duplicated original token (getUserData appears in both fields) wastes bytes and complicates relevance — the score boost from matching both fields is artificial.
BNGram tokenization (n=2..3) — index every 2-gram and 3-gram rejected Massive index size blowup (~10× payload). And it solves the wrong problem: NGrams enable substring search, but the user actually wants word-level search ("user" matching the word "User", not "use" matching the bigram "us" in "getUserData").
CEmbedding-only — drop the Text index, rely on dense vectors rejected Loses exact-match recall for short queries. BGE-small dense vectors are good at semantic similarity, not at "find the session whose title contains the literal substring getUserData". Two different jobs.
DEmbed Tantivy (full-text engine) in-process for the title field only rejected Adds a second indexing system, ~10 MB compiled. Just for one payload field. Operational cost ≫ benefit.
EMulti-value payload: ai_title_tokens = ["getUserData", "get", "User", "Data"] picked Qdrant Text index treats each array element as an independent token. No duplication of the original (it lives in ai_title alone). Native, semantically clean. Discussed below.
FUse Whitespace tokenizer + pre-split payload "getUserData get User Data" equivalent to A Same byte cost as A. Whitespace tokenizer doesn't change anything since the split is already done client-side.

Industry precedent — Best practice

  • Elasticsearch word_delimiter_graph token filter: the canonical solution for the camelCase/snake_case problem. Splits "Wi-Fi 4G" → ["Wi", "Fi", "4G"]; "PowerShot" → ["Power", "Shot"]. This is exactly the field-side transform we're doing client-side. Doing it pre-index in our Rust code vs. Elasticsearch's analyzer chain is the same operation, just at a different layer.
  • GitHub Code Search (per Ben Frederickson's writeup): tokenizes identifiers at indexing time, preserves both the original symbol and its constituent words. Same pattern as Option E.
  • Sourcegraph zoekt indexer: similar — pre-tokenized symbols stored alongside originals.
  • Qdrant docs on payload arrays: payload arrays of strings are first-class; Text index treats each element as a token. Confirmed in Qdrant 1.18.

Refined recommendation — Option E (multi-value payload)

// src-tauri/src/indexer.rs
fn identifier_tokens(s: &str) -> Vec<String> {
    let mut out: Vec<String> = Vec::new();
    for word in s.split_whitespace() {
        out.push(word.to_string());                   // preserve original
        out.extend(split_camel_case(word));           // ["getUserData"] → ["get","User","Data"]
        out.extend(split_snake_case(word));           // ["fix_dns_bug"] → ["fix","dns","bug"]
    }
    out.into_iter().filter(|t| t.len() > 1).collect() // drop noise tokens
}

// at point build (Qdrant payload values support Vec<String> natively):
let mut payload = HashMap::new();
payload.insert("ai_title".to_string(),        Value::from(title.clone()));
payload.insert("ai_title_tokens".to_string(), Value::from(identifier_tokens(&title)));

Qdrant's Text payload index on ai_title_tokens indexes each array element independently with the word tokenizer. A search for user now matches via the constituent "User" token without the byte-duplication cost of Option A. No ai_title duplication, no NGram blowup, no upstream wait.

Δ from v1: array payload (E) is strictly better than string concat (A) — same retrieval power, ~half the storage cost, clean semantics. v1 chose A because the multi-value payload pattern wasn't explicitly checked against the Qdrant docs.

03Issue #15 — Widen indexer::LensWeights

issue #15

Restated problem

Two parallel LensWeights structs (indexer:: = 6 fields; lens:: = 8 fields with diversity + fusion) connected by a lossy From impl. Web API callers can't request MMR or RRF because those fields don't exist on the public struct. The synchronization bug class that PR #12 REV-8 (CRITICAL) caught was an instance of this drift.

Alternatives matrix

#OptionVerdictRationale
AAdd 2 fields to indexer::LensWeights + make From lossless considered (v1 pick) Keeps both structs but makes them mirror exactly. The problem: nothing structurally prevents future drift. Same class of bug REV-8 fixed can reappear when fields #9, #10 land.
BExtension struct: LensWeightsAdvanced { diversity, fusion } as separate optional JSON object rejected Forces clients to think about "basic" vs "advanced" knobs, which isn't a real distinction — diversity and fusion are no more "advanced" than per-vector weights. API confusion ≫ benefit.
CTrait-based abstraction (trait LensConfig) rejected Trait abstraction makes sense when there are ≥2 concrete impls. We have one. Adding a trait for a single impl is Java-disease.
DMove LensWeights to a new shared types module, both modules import it considered Single source of truth ✓. But still requires moving 3 things (FusionMode, LensWeights, related helpers) into a new module — pure mechanical churn.
EDelete indexer::LensWeights; pub use crate::lens::LensWeights at indexer level picked Single struct, period. The drift bug class is eliminated by construction, not by discipline. pub use at the indexer module preserves the public path indexer::LensWeights for backward source-compat — callers don't need to change their import paths.
FBuilder pattern with optional setters rejected Builders shine when there are many fields with defaults and complex validation. Our struct has 8 fields with simple defaults; serde already does this. Builder is over-engineering.

Industry precedent — Best practice

  • Rust API Guidelines — C-COMMON-TRAITS: prefer one canonical type with re-exports over parallel types. rust-lang.github.io/api-guidelines.
  • serde with #[serde(default)] on all optional fields: backward-compatible JSON evolution without breaking old clients. The struct gains fields; old JSON deserializes by filling defaults.
  • tokio / axum re-export pattern: types live in one canonical crate, re-exported at the API boundary. E.g. axum::body::Bytes is a re-export of bytes::Bytes. Single source of truth, multiple stable import paths.
  • Stripe's API field-deprecation cycle: when fields move, the old path returns the same value via a re-export / alias for ≥1 release cycle.

Refined recommendation — Option E (single struct via re-export)

// src-tauri/src/lens.rs (unchanged — keeps the canonical struct)
pub struct LensWeights {
    pub content: f32, pub tool: f32, pub path: f32, pub error: f32, pub code: f32,
    pub content_late: f32, pub diversity: Option<f32>, pub fusion: FusionMode,
}

// src-tauri/src/indexer.rs (replace the duplicate struct)
pub use crate::lens::{LensWeights, FusionMode};
// every existing `crate::indexer::LensWeights` path still works.

What this kills: the entire drift bug class. REV-3, REV-8, REV-15 were all instances of the same root cause — two structs that have to stay in sync. After this change there is one struct.

Knock-on simplifications: impl From<crate::indexer::LensWeights> for LensWeights in lens.rs:869 is deleted (identity conversion is trivial). The "Adapter" comment block is deleted. The serde-default mirror requirement disappears.

Δ from v1: same fields, but instead of mirroring the struct we collapse to one. Cleaner blast radius, fewer future-PR hazards. The frontend changes (2 new controls) and TypeScript SDK regen are unchanged.

04Issue #16 — memex_points_indexed_total + MCP path

issue #16

Restated problem

The counter is exposed at /metrics and incremented correctly from the desktop indexing path. The MCP path doesn't increment it — because every one of the 11 MCP tools is read-only by design. Whether that's a bug depends entirely on what you think "MCP path" should look like.

Alternatives matrix

#OptionVerdictRationale
ATODO comment + tracking issue (Stage 1) + close-when-MCP-write-tool-ships (Stage 2) picked (v1 + v2) Honest: the metric is correct as-is; no current bug; tracks the wire-up so it's not forgotten when MCP grows a write surface.
BRemove the counter entirely (since it's always 0 in MCP-only deployments) rejected — anti-pattern Always-zero counters communicate "this dimension exists; it is just not active here". Removing the line makes the metric set look different across deployments (desktop vs server) when it should be uniform. See practice citation below.
CCompile-time feature gate (#[cfg(feature = "mcp-write")]) rejected Pre-emptively gates a feature that doesn't exist. Two binaries to maintain, two test paths, for zero current functional difference. YAGNI.
DOpenTelemetry semantic-convention-based generic instrumentation future work Valuable when we have ≥3 trace consumers. We have 1 (/metrics scrape). Premature.
EAdd a no-op MCP write tool just to exercise the path rejected Adding API surface to test instrumentation is the tail wagging the dog. Test the instrumentation when the real tool ships.

Industry precedent — Best practice

  • Prometheus best practices: "Counters which are always zero are valid measurements — they communicate that the dimension exists in the metric set." prometheus.io/docs/practices/instrumentation. Removing them creates dashboard / alerting fragility across deployment variants.
  • Google SRE Book — Monitoring chapter: "Instrument user journeys, not just code paths. A journey may be dormant; its instrumentation should still be present." (Chapter 6, "Monitoring Distributed Systems".)
  • Honeycomb "instrument speculatively": add the counter when the code path is plausible, not when it's active. The cost of an idle counter is bytes; the cost of missing instrumentation when you need it is incident-time blindness.
  • OpenTelemetry semantic conventions: db.operations_total{operation="insert"} is expected to be present on read-only deployments and read zero. Validates the same principle in the OTel world.

Refined recommendation — Option A (Stage 1 today, Stage 2 conditional) with explicit justification

Same plan as v1, but the v1 issue body framed it as "tracking issue, not a fix". v2 frames it more precisely: the counter as it stands is the correct implementation of a Prometheus best-practice pattern, not an instrumentation gap waiting for a fix.

The Stage 1 wire-up is unchanged:

// src-tauri/src/mcp.rs
//
// TODO(metric, see Issue #16): if/when an MCP write tool (e.g. `index_session`,
// `enrich_session`, `upsert_payload`) is added, increment
// `state.metrics.mark_indexed(n)` in the handler. The counter is already
// declared and emitted at /metrics — only the increment site is missing.

Δ from v1: no behavior change, but the issue body and this report now cite Prometheus's own guidance so a future reader doesn't see "always-zero counter" and "delete it as cleanup".
