How a memory surfaces — the whole mechanism

Three deterministic steps. There is nothing else hiding behind them. The "it knew what I needed" feeling is just nearest-neighbor lookup in a shared vector space.

STEP 1 · ONCE

Your history is embedded

Past Claude Code / Codex sessions (plus legacy transcripts and history.jsonl) become vectors and are stored as points in Qdrant. This is indexing — it happens up front, not while you read.

STEP 2 · LIVE

Your current context becomes a point

What you're doing right now is turned into a vector too — in the same space as everything you've ever done. Same input always lands at the same coordinates.

STEP 3 · LIVE

The nearest points are returned

Qdrant returns the closest stored points by distance. Those are the memories that "come back." Run it twice on the same context and you get the same result, in the same order. No sampling, no guessing.

No LLM runs at retrieval time. Embedding plus vector search is the entire pipeline that brings a memory back. Nothing rephrases your past, invents a summary, or "thinks" on your behalf. The structure of your own history is what surfaces it — Vannevar Bush's 1945 associative trails, made literal as distance.

Run it locally, or wire it into your agent

Your sessions stay a corpus you own — Claude Code silently migrates and deletes old ones; Memex keeps them and lets you move by similarity.

Add the MCP server (11 tools over stdio + HTTP)

# register Memex as an MCP server
claude mcp add memex \
  -- memex-mcp --transport stdio

# or run the all-in-one server / Docker variant
# (Qdrant + web UI + MCP in one image)
docker run -p 8080:8080 \
  -v ~/.claude:/data/claude:ro \
  twoweeks/memex

Works in Claude Code, Codex, and Cursor. Exact commands & flags live in the repo README.

What you can rely on

  • Deterministic recall. Same context returns the same points, same order.
  • LLM-free at runtime. Embed + vector search only. No chatbot, no generation.
