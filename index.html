What it is

An associative memory for your AI work — not another chatbot.

Memex reads ~/.claude/projects/**/*.jsonl and ~/.codex/sessions/**/*.jsonl, embeds every session into five named vectors, and stores them in a local Qdrant collection. Then it gives you spatial surfaces — sliders, a topology galaxy, mix & match, proactive recall, and an agent-memory layer — to arrive at the right past moment, not search for it.

🜨Local-first & LLM-free at runtime

One embedder (BGE-small, 384-d) runs once at index time. Every surface — sliders, MST, recall, Wrapped — is deterministic Qdrant math on a local store. Zero network calls during use. Your session text never leaves your laptop.

⟁Associative, not interrogative

A chatbot waits for a query. A memex notices the moment. SessionStart injects a Companion primer; a repeated-error pattern surfaces a Loop Breaker; the topology galaxy lets you drift between adjacent sessions by named-vector affinity. The visitor is a recipient, not an operator.

The geometry

Five named-vector lenses on the same conversation.

Every session becomes one point in Qdrant with five orthogonal embeddings — five ways of asking "what is this session about?". Weight any subset and Qdrant fuses them in a single Universal Query. No RRF, no two-round hack: real multi-vector scoring with continuous weights.

content · 384-d

What was said

The semantic core — user intent, assistant reasoning, project topic. The default lens; drop it to 0 to feel what the others see.

tool · 384-d

Which tools fired

The shape of the agent's action — Bash, Edit, Read, WebFetch — independent of words. Surfaces tool-shape twins.

path · 384-d

Which files were touched

The repo-coordinate footprint. Find sessions that worked the same code paths, even if the prose was unrelated.

error · 384-d

What broke

Built from only the error spans. The recall surface keys on this lens to ask: "have I solved this exact failure before?"

code · 384-d

Which code shapes

Identifier and snippet density — function names, types, APIs. Surfaces sessions that wrote near the same code, even across projects.

The map

Topology — your corpus as a minimum-spanning galaxy.

Every session is a node, colored by project. Edges connect each session to its nearest content-vector neighbor — a Minimum Spanning Tree over cosine distance. Drift along the edges and you walk the conceptual neighborhood, not the timeline.

Drift, don't search

A timeline tells you when. A topology tells you near what. The MST keeps the graph connected and sparse — every node has at least one neighbor, no node is more than a handful of hops from any other.

Click an edge and you've crossed a conceptual border in one move. Two seemingly unrelated projects often turn out to share an edge — and that edge is usually the insight you forgot you had.

The arithmetic

Mix & match — interpolate between two moments, get the third.

Vector arithmetic on session embeddings. Pick two sessions A and B, slide a single knob, and Qdrant returns the sessions that live on the line between them. Discover overlap you didn't know you had.

A + B linear

Combine two sessions' content vectors with a 50/50 weight and search. Surfaces sessions that share the intersection of both topics — not the union.

mix_similar_sessions(a, b, weight=0.5)

A − B delta

What's in A but not in B. Subtraction in vector space surfaces "this session, minus that obvious context" — useful for de-duping the same project from another angle.

mix_similar_sessions(a, b, weight=-1.0)

(A + B) / 2 centroid

The conceptual midpoint of a small cluster — feed three or four sessions, get the centroid, and find the "average" of that idea-cloud. Where in your past corpus does that idea already live?

centroid(sessions[]) → search

The reflex

Proactive recall — the error lens reaches for you.

Every ~12 seconds the desktop app polls tail_recent_errors(). On a fresh error, it embeds the error text with BGE-small and queries the dedicated error named vector with a has_errors = true payload filter — asking, on your behalf, "have I solved a similar failure before?". If yes, the next-action hint surfaces as an OS notification before you ask.

# a fresh agent error just landed in this session Error: failed to connect to Qdrant gRPC at :6334 (connection refused) # memex polls every ~12s, embeds the error, queries the `error` named vector → match project-x · 4 days ago · solved: docker compose up qdrant → match project-y · 11 days ago · solved: started qdrant container first # OS notification fires — past-you's next action, surfaced on its own

The dedicated error lens earns its keep

The error vector is built from only the error spans — not the surrounding chat. So "connection refused on :6334" doesn't get drowned out by the user's pleasantries. The payload filter has_errors = true keeps the search inside sessions that actually broke.

Five sliders, five lenses — but recall is the one that moves first. The proactive loop is what makes a memex feel alive.

The agent layer

Companion · Wrapped · Loop Breaker — memory in the agent's turn.

Three surfaces that put Memex into the model's context window without a plugin — through host hooks and MCP. Each is gated to stay quiet until it has something concrete to say.

🜨 Companion SessionStart hook

A 200–500 token primer at session start — past intents, decisions, known pitfalls in this cwd — composed deterministically from the corpus, no LLM. The cold-start killer.

compose_memory_primer(cwd) → injected context

🜚 Wrapped CLI · MCP tool

An engineering "Spotify Wrapped" of your corpus — top tools, top file extensions, intent / arc / outcome distribution, repeated decisions ("I keep re-deciding the same thing"), debugging fingerprint, cross-agent Claude ↔ Codex split. Pure payload aggregation, deterministic.

generate_wrapped_report(window_days=30) → markdown

🜉 Loop Breaker PostToolUse hook

After every Bash tool call, a tiny watcher looks at the last few turns. On a repeated-error stuck pattern, it surfaces "what past-you ran next" — a single pivot, not a wall of text. Fail-open: if the engine's down, your session continues without it.

loopcheck(last_turns) → notify(pivot)

Hooks live in your gitignored .claude/settings.local.json — never committed. Inputs are redacted for tokens / keys / private blocks before they ever touch the index. See the full guide → docs/agent-integration.md.

Qdrant — every feature we lean on

The Qdrant techniques behind each surface.

Memex is built on six concrete Qdrant 1.18 primitives — not just "vector search". Each surface above maps to a specific server-side feature that a single-vector setup couldn't reproduce. Code pointers are real; click any to read the call site.

Q1

Named vectors per point

What it is. One Qdrant point holds five named dense vectors — content, tool, path, error, code, each 384-d cosine BGE-small — stored as a single VectorParamsMap. Five orthogonal lenses on the same conversation, each with its own binary-quantized HNSW index.

Why it matters. A single-vector store forces you to pick one "meaning". Five named vectors let us ask "what tools fired?" and "what files were touched?" as independent questions on the same point.

// PointStruct: one id, many named vectors, one payload
PointStruct {
  id: uuid_v5(session_id),
  vectors: NamedVectors { content, tool, path, error, code },
  payload: { project, branch, has_errors, source_path, … }
}

Q2

Universal Query — weighted multi-vector scoring

What it is. The lens slider issues one Universal Query per non-zero weight via QueryPointsBuilder.using("<name>"), then blends per-vector scores with continuous weights. Not RRF: rank-based fusion ignores weights, so we compute Σ(scorei · wi) / Σ wi ourselves.

Why it matters. "Weight tool calls 2× more than prose" is a real, continuous knob. RRF can't say that.

QueryPointsBuilder::new("memex_sessions")
  .query(embedded)
  .using("content")   // or tool / path / error / code
  .limit(20)
  .with_payload(true)

indexer.rs::lens_search

Q3

Discovery API — context pairs for mix & match

What it is. DiscoverInput { target, context: ContextInput } wrapped in a Query: each ContextInputPair { positive, negative } is a discrimination hint Qdrant uses to re-rank around target. Mix & match → "find sessions like A and B, biased against C".

Why it matters. Plain recommend ("more like Y") and plain search ("near Q") can't express "find X — but only the kind that isn't like Z". Discovery API does, server-side, in one round-trip.

Query::Discover(DiscoverInput {
  target:  positive_a,
  context: ContextInput {
    pairs: [
      ContextInputPair { positive: a, negative: c },
      ContextInputPair { positive: b, negative: c },
    ],
  },
})

Q4

Distance Matrix → MST (server-side K-NN graph)

What it is. search_matrix_pairs with .using("content").sample(N).limit(K) returns top-K nearest neighbors per sampled point — a sampled K-NN graph in one round-trip. We feed pairs into petgraph::UnGraph and compute the minimum spanning tree client-side for the galaxy.

Why it matters. Most vector DBs make you do N² queries client-side or expose only batch search. Qdrant ships the K-NN graph as a first-class endpoint — exactly what an MST renderer needs.

SearchMatrixPointsBuilder::new("memex_sessions")
  .using("content")
  .sample(N)   // points to consider
  .limit(K)    // neighbors per point
// → SearchMatrixPairs { pairs: [(a, b, score), …] }
// → petgraph::min_spanning_tree

indexer.rs::topology · qdrant-features.md §3

Q5

Payload filter as cheap HNSW pre-filter

What it is. Proactive recall searches the dedicated error named vector while passing a keyword/bool filter on the has_errors payload field. Because the field has a Qdrant field index (CreateFieldIndexCollectionBuilder), the filter runs as a cheap HNSW pre-filter — not a full collection scan.

Why it matters. "Have I seen this error before?" needs to ignore happy-path sessions whose text happens to overlap. The dedicated error vector + the indexed payload predicate make recall feel instant on 1000s of points.

QueryPointsBuilder::new("memex_sessions")
  .query(embedded_error)
  .using("error")              // the dedicated lens
  .filter(Filter {
    must: vec![ Condition::matches("has_errors", true) ],
  })
  .limit(5)
  .with_payload(true)

Q6

Native snapshots — HTTP create / download / restore

What it is. Qdrant exposes built-in collection snapshots over HTTP. Memex wraps the three endpoints with reqwest so a corpus is a single portable file — move it between machines, archive it, sign it. Embeddings + payload travel together; raw JSONL stays on disk.

Why it matters. No external object store, no ad-hoc dump format. The collection IS the artifact.

// create + filename
POST /collections/{c}/snapshots
// download bytes
GET  /collections/{c}/snapshots/{name}
// restore
POST /collections/{c}/snapshots/upload?priority=snapshot

indexer.rs::snapshot_export / snapshot_import · qdrant-features.md · Snapshot

And the small things that make it fast

  • Deterministic point ids. point_id = uuid_v5(session_id) — upserts are idempotent; rebuilds don't duplicate.
  • Binary-quantized HNSW per named vector. Each of the five lenses has its own index — recall & latency tuned per lens, not per collection.
  • Field-indexed payload predicates. has_errors (bool), project, branch are indexed via CreateFieldIndexCollectionBuilder — filters short-circuit before HNSW walk.
  • Server-side FormulaQuery. Prefetches carry per-vector recency decay, applied server-side — no client round-trip to re-score.
  • Payload kept lean (≤1 KB). Only metadata + source_path; the transcript itself stays in JSONL on disk and is re-parsed on demand for replay.
  • Two transports, one client surface. gRPC (qdrant-client 1.18) for queries; HTTP (reqwest) for snapshot endpoints. Same collection, same data.

The fine print, up front

Local-only, redacted at the door, fail-open under load.

🛡 Secret redaction at index time

Session JSONL frequently contains tokens, API keys, PEM blocks. Memex's redact.rs neutralizes Slack / GitHub / OpenAI / AWS keys and PEM payloads before embedding, so the vector store never sees them. The agent-integration package ships the same redactor for hook inputs.

🪶 Fail-open hooks

If the Memex engine isn't running, the SessionStart / PostToolUse / SessionEnd hooks return empty and the agent's turn proceeds with no memory — rather than stalling. Your agent never blocks on Memex.

🔒 Loopback by default

The all-in-one server binds 127.0.0.1:8765. Exposing it on the LAN is an opt-in. The Docker image runs as a non-root user and includes a healthcheck.
